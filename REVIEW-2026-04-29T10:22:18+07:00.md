# Production-ready bar for this PR

- The GVS `allowBuilds = {}` default must not interfere with restoring `dangerouslyAllowAllBuilds` from global config.yaml or reading `allowBuilds` from the global workspace manifest
- The precedence order must be: workspace manifest `allowBuilds` > global config `dangerouslyAllowAllBuilds` > GVS default `{}`
- The guard must use explicit null/true checks, not `hasDependencyBuildOptions()`, because `globalDepsBuildConfig` re-applies `dangerouslyAllowAllBuilds: false` (which is `!= null`)
- Tests must prove the behavior for both policy paths (workspace manifest + dangerouslyAllowAllBuilds) with GVS enabled
- Tests must clean up filesystem and environment side effects
- PR title and description must accurately describe the fix scope (not ".npmrc")
- Changeset must describe the root cause and fix accurately
- No stale or misleading comments

# Findings

## NIT PR title inaccurately references ".npmrc"
- Type: Verified issue
- Evidence: PR title "fix: global installs respect .npmrc build policy when GVS is enabled"
- Why it matters: The fix is about `dangerouslyAllowAllBuilds` from global config.yaml and `allowBuilds` from the workspace manifest — not .npmrc. `loadNpmrcConfig()` does not read build policy keys. Misleading title could confuse reviewers and future readers.
- Recommendation: Update PR title to "fix: global installs respect build policy from global config.yaml when GVS is enabled"
- Confidence: High

## NIT Changeset references ".npmrc" implicitly via "dangerously-allow-all-builds" kebab-case
- Type: Verified issue
- Evidence: `.changeset/fix-global-allow-builds.md:6`: "global config build policy (e.g., `dangerously-allow-all-builds`)"
- Why it matters: The kebab-case `dangerously-allow-all-builds` is the .npmrc key name; the YAML config uses `dangerouslyAllowAllBuilds`. Minor but could confuse about which config file is involved.
- Recommendation: Change to `dangerouslyAllowAllBuilds` (camelCase) in the changeset since it's read from config.yaml, not .npmrc
- Confidence: High

## NON-BLOCKING First test doesn't assert enableGlobalVirtualStore is true
- Type: Verified issue
- Evidence: `config/reader/test/index.ts:1715-1762`
- Why it matters: The test's premise depends on GVS being enabled, but doesn't assert it. If the default changes, the test would silently test the wrong path.
- Recommendation: Add `expect(config.enableGlobalVirtualStore).toBe(true)` like the second test does at line 1792
- Confidence: High

## NON-BLOCKING First test asserts dangerouslyAllowAllBuilds: true but this is a side effect, not the primary behavior under test
- Type: Verified issue
- Evidence: `config/reader/test/index.ts:1749`: `expect(config.dangerouslyAllowAllBuilds).toBe(true)`
- Why it matters: This assertion verifies that config.yaml was read correctly, but the test is about workspace manifest precedence. The `dangerouslyAllowAllBuilds: true` persists because `addSettingsFromWorkspaceManifestToConfig` reads config.yaml AFTER `extractAndRemoveDependencyBuildOptions` strips it, re-setting it on the config object. This coexistence is correct behavior but is an architectural side effect that makes the test slightly confusing about what's being proven.
- Recommendation: Add a comment explaining why this assertion exists (verifying config.yaml was read, not overwritten by the GVS default)
- Confidence: Medium

No issues found in these areas based on the diff and reviewed context:
- **Correctness & functional completeness**: The fix correctly moves the GVS default after workspace manifest reading and globalDepsBuildConfig re-application. The guard `allowBuilds == null && dangerouslyAllowAllBuilds !== true` is correct because `hasDependencyBuildOptions` was always true after `globalDepsBuildConfig` re-applied `dangerouslyAllowAllBuilds: false`.
- **Architecture & boundary integrity**: The change stays within the config reader. No cross-boundary violations.
- **Code clarity**: The comment at lines 495-500 is accurate and explains intent well.
- **Performance**: One additional null check and one boolean comparison — negligible.
- **Tests & validation**: Both critical paths are covered (workspace manifest precedence + dangerouslyAllowAllBuilds-only). Cleanup is present for both filesystem and env vars.
- **Operational risk**: Low. The guard is conservative — only applies `{}` when both `allowBuilds` and `dangerouslyAllowAllBuilds` are absent/non-true.
